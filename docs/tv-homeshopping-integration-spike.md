# TV홈쇼핑 채널 연동 가능성 조사 (ARK-15 스파이크)

**상태:** 조사 완료 · **코드 구현 없음(스코프 밖, 이슈 정의대로)** · **채널 우선순위는 CEO 결정 대기**
**대상:** GS샵(舊 GS홈쇼핑), CJ온스타일, 현대홈쇼핑, (참고) 롯데ON — TV홈쇼핑/T커머스 벤더 연동
**소유자:** Founding Engineer · **이슈:** ARK-15 · **날짜:** 2026-07-01

이 문서는 보드 결정(2026-07-01, "채널 확대 2순위, 오픈마켓과 별도 트랙")에 따른 조사
결과다. 결론부터: **TV홈쇼핑은 "API 있음/없음"으로 한 줄 정리가 안 되고, 채널마다
다르다.** 그래서 이 문서는 (1) 채널별 연동 가능 여부, (2) 오픈마켓과 근본적으로 다른
점, (3) 어댑터 패턴이 그대로 먹히는지, (4) 예상 공수와 리스크, (5) CEO 결정이 필요한
지점 순으로 정리한다. **코드는 만들지 않았다** — 이슈 정의상 산출물은 이 문서이고, 실
구현은 결론에 따라 후속 이슈로 분리한다.

**핵심 결론:** "오픈마켓처럼 API 있음" vs "완전히 수동" 이분법이 아니라, **채널별로
개별 확인이 필요한 스펙트럼**이다. CJ온스타일은 오픈마켓과 유사한 API 연동이 가능해
보이고, GS샵은 현재 포털+엑셀 다운로드가 주된 경로로 보인다. 따라서 "TV홈쇼핑 어댑터"
하나를 일반화해서 만들 수 없고, **채널별로 API 유무를 먼저 확인한 뒤에만 어댑터 여부를
결정**해야 한다.

## 2. 오픈마켓과 근본적으로 다른 점

API 유무와 별개로, TV홈쇼핑은 주문 도메인 자체가 오픈마켓과 다르다. 이는 API가 있는
채널(CJ온스타일 등)이라도 `MarketplaceAdapter` 정규화 계약을 그대로 적용하기 전에
검토해야 할 지점이다.

1. **방송 편성 기반 주문 폭증(버스트).** 오픈마켓 주문은 시간에 걸쳐 완만하게 들어오지만,
   TV홈쇼핑 주문은 **방송 회차(보통 20~60분)** 동안 집중적으로 폭증한다. 네이버 어댑터가
   쓰는 "24시간 창 + 완만한 폴링" 모델(`docs/naver-commerce-integration.md` §3)은 그대로
   안 맞을 수 있다 — 방송 시간대에 맞춘 버스트 대응 폴링이 필요할 수 있다.
2. **콜센터/ARS 주문 비중이 매우 높다.** 업계 자료상 TV홈쇼핑 주문의 **60% 이상이
   ARS(전화 자동응답)** 를 통해 들어온다. 즉 주문이 앱/웹 체크아웃이 아니라 콜센터
   상담원·ARS 입력을 거쳐 생성되고, 이 처리 단계 때문에 방송 종료 후 **주문 데이터
   반영에 지연(lag)** 이 있을 수 있다. `NormalizedOrder.orderedAt`을 "방송 시각"으로 볼지
   "주문 확정 시각"으로 볼지 채널별로 확인 필요.
3. **상품이 "방송 회차" 단위로 관리되는 경우가 많다.** 사은품 구성, 가격, 옵션이 방송마다
   바뀌는 일이 흔해서, 오픈마켓의 "고정된 채널 상품ID" 가정(`NormalizedProduct.
   marketplaceProductId`)이 TV홈쇼핑에서는 "방송 회차ID + 상품ID" 복합키로 바뀔 수 있다.
4. **정산 구조가 다르다.** 오픈마켓 수수료 외에 방송 송출 관련 비용/프로모션 항목이 얽혀
   있어, `Settlement` 매핑(ARK-7 예정)은 TV홈쇼핑 전용 규칙이 추가로 필요할 가능성이 높다.
5. **반품/배송은 이미 3자 연동 계층이 존재.** goodsFLOW(굿스플로) 같은 배송추적/반품
   자동화 서비스가 현대Hmall·카카오몰 등 대형몰에 이미 붙어 있다 — TV홈쇼핑 벤더들도
   이런 물류 계층을 쓰는 경우가 많다. 배송 상태 동기화는 "채널 API"가 아니라 이런
   물류 중개 서비스를 통하는 경로가 현실적일 수 있다(추가 조사 필요, 이번 스파이크
   범위 밖).

## 3. 어댑터 패턴이 그대로 맞는가

**API가 있는 채널(CJ온스타일 등):** `MarketplaceAdapter` 계약(`src/integrations/
marketplace.ts`)이 큰 틀에서는 재사용 가능해 보인다. 다만 위 §2의 도메인 차이 때문에,
계약 자체를 손대기보다 **정규화 매퍼(mapper) 레벨에서 흡수**하는 편이 낫다 — 예:
`orderedAt`을 "방송 편성 시각"과 "주문 확정 시각" 둘 다 `raw`에 보존하고 대표값만
정규화 필드로 노출. 이는 네이버 어댑터가 이미 쓰는 패턴(원본 보존 + 정규화값 분리)과
같은 방향이라 계약 변경 없이 확장 가능할 것으로 본다.

**API가 없는 채널(GS샵 등, 확인된 한도 내):** 어댑터 패턴이 애초에 성립하지 않는다.
`fetchOrders`가 전제하는 "실시간 원격 호출"이 없기 때문이다. 대안은 두 가지이며, 코드
구현 전에 **CEO/보드 판단이 필요**하다(§5):

- **(a) 엑셀 임포트 경로.** 협력사가 포털에서 내려받은 주문 엑셀을 ARKAIN에 업로드하면
  파싱해서 동일한 `NormalizedOrder`로 변환. 낮은 리스크, 낮은 자동화 수준(셀러가 매번
  수동으로 다운로드/업로드해야 함). 오픈마켓과 별개의 "임포트 파이프라인" 컴포넌트로
  설계해야 하며, `MarketplaceAdapter`를 억지로 흉내 내지 않는 편이 낫다(계약을 "가짜
  fetchOrders"로 왜곡하면 다음 파일 기반 채널이 늘 때 오히려 헷갈린다).
- **(b) 포털 스크레이핑/RPA.** 자동화 수준은 높지만, 협력사 포털 로그인 자격증명을
  ARKAIN이 보관·사용해야 하고, ToS/약관 위반 소지가 있다. ARCHITECTURE.md §7의 "실 셀러
  자격증명은 CEO 명시 승인 없이 건드리지 않는다"는 경계에 정면으로 걸리는 데다, 일반
  API 자격증명보다 리스크가 크다(포털 UI 변경 시 깨짐, 계정 잠금 위험). **권장하지 않음**
  — 하더라도 별도 CEO 승인 + 리스크 검토가 선행되어야 한다.

## 5. CEO 결정이 필요한 지점 (boundary)

이 스파이크는 조사이고, 실 구현은 다음 결정 없이는 시작하지 않는다(AGENTS.md 경계:
"business/product 결정은 CEO에게").

1. **어느 TV홈쇼핑 채널을 먼저 붙일지.** 조사 결과 CJ온스타일이 API 존재가 가장
   확실하고 어댑터 패턴과도 가장 잘 맞는다 — **CJ온스타일을 1순위로 추천.** GS샵은
   API가 아니라 엑셀 임포트 성격이라 "채널 확대"라기보다 별개의 기능(파일 임포트
   파이프라인)에 가깝다.
2. **GS샵류(포털+엑셀) 채널에 "엑셀 임포트" 기능을 MVP 스코프에 넣을지.** 이건
   `MarketplaceAdapter` 확장이 아니라 신규 컴포넌트라서, 우선순위/공수 판단이 필요하다.
   (권장: 오픈마켓 어댑터 확장이 어느 정도 자리잡은 뒤 별도 이슈로.)
3. **포털 스크레이핑/RPA는 하지 않는 것을 기본값으로 제안.** ToS 리스크 + 자격증명
   경계 문제 때문이다. 필요하다고 판단되면 별도 리스크 검토 + 명시적 사인오프 요청.
4. **입점/계약 절차.** CJ온스타일 API 신청은 "입점 협력사"만 가능하다 확인됐다 —
   즉 실제 스토어(또는 테스트 협력사 계정)가 없으면 API KEY 자체를 못 받는다. 이는
   네이버 스파이크(ARK-3)와 같은 종류의 블로커이며, 여기도 CEO가 풀어야 하는 지점이다
   (테스트 협력사 계정/계약 여부 확인).

## 6. 조사 방법 및 한계

웹 검색 + 공개 페이지 열람으로 진행했다(실제 계약/로그인 없이 접근 가능한 공개
정보 한정). 따라서:
- CJ온스타일 API의 **실제 엔드포인트 스펙**(요청/응답 필드, 페이징 방식, 레이트리밋)은
  로그인 후 문서에서만 확인 가능해 이번 조사에는 포함되지 않았다 — 입점 계약 이후
  1차 스파이크에서 확인해야 할 항목이다.
- GS샵 "파트너스"가 정말 API를 전혀 제공하지 않는지는 100% 확신할 수 없다 — 공개된
  페이지에서 API 문서 링크를 찾지 못했다는 것이 근거이며, 협력사 로그인 후에만 보이는
  메뉴가 있을 가능성은 남아있다. **결론이 아니라 현재까지의 최선 추정으로 취급할 것.**
- 롯데홈쇼핑/NS홈쇼핑/홈앤쇼핑/공영홈쇼핑은 이번 스파이크에서 상세 조사하지 않았다
  (우선순위: CJ온스타일 > 현대홈쇼핑 확인 후 필요시 추가 스파이크).

## 7. 다음 단계 제안

이 문서 자체가 이슈의 산출물이므로 ARK-15는 조사 완료로 처리한다. 코드 구현은 위 §5의
CEO 결정에 따라 후속 이슈로 분리 제안:
- CEO가 CJ온스타일을 1순위로 확정하고 입점/계약 상태가 확인되면 → "ENG-CJOnstyle-Spike"
  (ARK-3 패턴과 동일하게 어댑터 구현 + 자격증명 블로커 문서화).
- GS샵 엑셀 임포트를 MVP 스코프에 넣기로 하면 → 별도 이슈로 분리(어댑터가 아닌 임포트
  파이프라인으로 설계).
